# 03. 에이전트 협업 기록 — 초기 구조 설계

작성일: 2026-04-23

요약: 초기 프로젝트 구조(디렉터리·컴포넌트 분리 등)를 설계할 때 코딩 에이전트를 어떻게 활용했는지 기록합니다.

## 사용한 에이전트 도구

- 코딩 에이전트(대화형): 구조 제안, 파일 템플릿 생성
- 컨텍스트 읽기: `read_file`, `semantic_search`
- 편집·적용: `apply_patch`
- 작업 추적: `manage_todo_list`

## 에이전트에게 위임한 작업의 범위와 그 판단 근거

- 위임한 작업:
  - 디렉터리 및 파일 구조 대안 제시
  - 컴포넌트·페이지 경계 초안(파일 템플릿 포함)
  - 네이밍 규칙과 기본 README 스켈레톤 생성
- 판단 근거:
  - 구조 설계는 여러 대안을 빠르게 비교하는 작업으로 에이전트가 다양한 옵션을 빠르게 생성할 수 있음
  - 규칙 기반 반복 작업(템플릿 생성 등)은 자동화에 적합
  - 최종 아키텍처·보안·비즈니스 규칙은 사람의 판단으로 확정

## 에이전트의 산출물을 검증한 방법

- 코드베이스 컨텍스트와 충돌 여부 수동 검토
- 타입/빌드 체크(예: `tsc`, `vite build`)로 오류 확인
- 변경된 구조를 적용해 주요 페이지 로드 여부를 수동 확인
- 팀 코드 리뷰로 최종 승인

## 에이전트가 잘못된 방향을 제시했을 때 교정 방법

- 근거 요청: 설계 근거(왜 이렇게 분리했는지)를 묻고 맥락을 보강
- 프롬프트 정제: 요구사항·제약(접근성, 성능, 라이브러리 제약)을 명시해 재요청
- 롤백 및 수동 보완: 변경이 부적절하면 되돌리고 수동으로 재구성

---

메모: 설계 단계에서는 에이전트 제안을 '입력'으로 사용하고, 사람 판단을 통해 최종 결정을 내리는 것이 안전합니다.
